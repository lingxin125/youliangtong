为什么分两段走

先抓进 Figma，再从 Figma 回流代码
HTML 到 Figma 解决的是“已有页面快速进入设计空间”；Figma 到代码解决的是“把设计上下文落成项目内可维护代码”。 两段目标不同，所以不要混着做。
如果直接从 `00 Raw Capture` 回流代码，容易把原型里的临时样式、演示结构、重复块一起带进正式工程。 最优路径是：Raw Capture 只做底稿，真正给代码的是 `01 Candidate Components` 或 `02 Polished Screens`。
最推荐
Figma 整理后，再做设计到代码
不要直接做
从未经清理的 Raw Capture 直接生成生产代码
最佳产物
组件化代码 + 映射关系 + 可持续迭代链路
2

标准流程

从 Figma 到代码的主线
1. 先把页面整理干净
UI 在 Figma 中确认正式视觉、层级、文案、状态和组件拆分。尽量不要从 Raw Capture 直接起代码。
2. 选中要实现的 node
优先选页面级 frame 或组件级 node，拿到 node URL、fileKey、nodeId。
3. 用 `get_design_context` 获取设计上下文
这是主工具，会返回截图、参考代码、上下文元数据、资产下载地址，适合做真正的设计到代码。
4. 先复用设计系统，再写业务代码
如果项目已有按钮、表单、卡片、弹窗组件，优先复用。缺失时再新建最少量组件。
5. 用 Code Connect 建立映射
关键组件一旦稳定，建议把 Figma node 和代码组件绑定起来，后续改版效率会大幅提升。
6. 回到仓库验证
检查视觉一致性、响应式、交互态、资产引用、可访问性和已有业务逻辑是否被破坏。
最稳的起点
02 Polished Screens
如果页面已经完成 UI 精修，这里最适合直接让 Codex 做页面实现。
组件级起点
01 Candidate Components
如果要先搭组件库、表单控件、卡片体系，这里比整页 node 更适合。
最不建议直接用
00 Raw Capture
这里只适合作为参考底稿，不适合直接生成正式代码，除非只是快速演示或做一次性草稿。

4

MCP 工具分工

设计到代码主工具
`get_design_context`
主工具。返回截图、参考代码、上下文元数据、资产下载信息。做页面或组件实现时优先用它。
`search_design_system`
在设计系统库里找组件、变量和样式，先查复用机会，再决定是否新建组件。
`get_variable_defs`
补颜色、字号、间距等变量信息，帮助把设计 token 对齐到代码。
长期维护工具
`get_code_connect_suggestions` + `send_code_connect_mappings`
给稳定组件建立 Figma 和代码的映射关系，后续 UI 与研发都能更快定位同一组件。
`get_code_connect_map`
查看某个 node 现在绑定了哪个代码组件，适合巡检和排错。
`use_figma`
用来写回 Figma，例如整理 page、批量改名、补注释、更新描述，不是生成代码主工具。

Prompt 模板

模板 A · 从单个页面 frame 实现页面代码
适合 React / Vue / HTML 页面实现
请使用 Figma MCP 读取这个 Figma 页面 node，并在当前仓库中实现对应页面代码。 要求： 1. 优先使用 `get_design_context` 2. 优先复用当前项目已有组件、样式约定和设计 token 3. 不要机械照搬截图，要按项目现有结构落地 4. 如需新建组件，只创建最少必要的部分 5. 产出后告诉我： - 修改了哪些文件 - 页面结构如何拆分 - 哪些地方是按设计做了工程化调整 Figma 页面链接： 【替换为具体 node URL】 目标技术栈： 【例如 React + Tailwind / Vue + CSS / 原生 HTML】
模板 B · 先做组件，再拼页面
适合 Figma 中已有 Candidate Components
请先读取这个 Figma 组件节点，先在仓库里实现对应组件，再把它接入页面。 要求： 1. 组件 props 要明确 2. 复用项目中已有按钮、文本、表单或布局组件 3. 样式尽量贴近设计，但保持可维护 4. 页面只做最少量接入改动 Figma 组件链接： 【替换为组件 node URL】 接入页面： 【替换为页面路由或文件】
模板 C · 设计变更后的增量更新
适合已有页面已上线，只想同步局部视觉改动
请根据最新的 Figma node 更新现有页面代码。 要求： 1. 只做与这次设计变更相关的最小改动 2. 不要重写整个页面 3. 优先复用已有组件 4. 输出时说明： - 哪些地方变了 - 哪些地方保持不动 - 是否存在设计与现有业务逻辑冲突 Figma node： 【替换为最新 node URL】 已有代码位置： 【替换为文件路径】
模板 D · 建立 Code Connect 映射
适合稳定组件，例如 Button / Card / Modal
请读取这个 Figma 组件 node，并为它建立 Code Connect 映射到当前仓库的真实组件。 要求： 1. 先获取建议映射 2. 映射前确认组件名称和代码路径 3. 如果需要，补充 template / props 说明 4. 完成后告诉我映射到了哪个组件 Figma 组件 node： 【替换为 node URL】 目标代码组件： 【例如 src/components/Button.tsx】
模板 E · 需要明确输出契约时
适合交给研发执行或做多人协作
请基于这个 Figma node 生成对应代码，但必须遵守以下输出契约： 1. 先说明准备修改的文件 2. 实现时优先复用现有组件和样式 token 3. 完成后必须返回： - 文件修改列表 - 视觉实现说明 - 响应式与状态说明 - 未覆盖风险 Figma node： 【替换为 node URL】 仓库要求： 【替换为技术栈与目录规则】

Code Connect 回路

建议建立这条长期链路
先把高频组件稳定下来
按钮、输入框、卡片、筛选条、弹窗、底部操作栏这类组件优先建立映射。
生成映射建议
先看建议是否符合代码现状，避免把一个设计 node 映射到错误组件。
保存映射
确认组件路径、名称、标签后再落库，后续团队成员可以直接复用这条对应关系。
设计变更时复用同链路
页面改版不用从零猜组件，而是从 Figma 映射直接找到仓库组件，迭代成本最低。
什么时候值得做 Code Connect
一个组件会被 3 个以上页面复用。
UI 和研发会在同一个组件上长期反复迭代。
你已经明确组件代码路径，不会频繁重命名目录。
如果页面还处在快速探索期，先不要急着建立映射；先做设计定稿，再建立映射更省事。

交付检查

代码是否优先复用了现有组件和设计 token，而不是复制一坨静态样式。
页面是否覆盖了 Figma 中的关键状态，而不是只还原默认态。
图片、图标、字体、颜色变量是否已落成项目可维护资源。
关键组件是否值得建立 Code Connect 映射，以支持后续持续改版。
