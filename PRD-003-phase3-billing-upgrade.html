1. 综述 (Overview)

1.1 项目背景与核心问题

丰收云（邮粮通）是面向农产品交易场景的微信小程序，服务收购商（含游商/收粮贩子）和农户两类核心角色。当前已上线的开单功能存在以下核心问题：

  1. 开单流程过于线性：当前开单页只有"立即付款"和"稍后结算"两个出口，无法覆盖真实收购现场的多种场景（挂单、待开价、欠账、转客户付款）。
  2. 缺少协作能力：游商（收粮贩子）替货主（客户）收粮后，无法将订单转给客户付款，只能线下沟通，资金流转依赖微信/银行卡外部转账。
  3. 缺少团队管理：收购商无法管理团队成员（开单员），开单员开的单无法归属团队统一管理。
  4. 开单页信息密集：操作人员年龄偏大，当前页面承载过多决策（付款方式选择等），导致使用门槛高。

竞品对标：商易通（17,974 MAU）已具备挂单、待开价功能，但无系统内付款能力。丰收云依托邮政金融的支付能力，三期补齐开单深度后可形成"专业开单 + 系统内付款 + 拍照留存"的差异化壁垒。

V2.0 更新说明（2026-04-18）：基于原型实现和多轮评审反馈，本版本做了以下关键调整：① 开单页底部改为「草稿箱图标 + 暂存按钮 + 立即付款按钮」三段式布局，取消组合弹层；② 新增统一订单详情编辑页（buyer-order-edit.html），替代跳回开单页编辑的旧方案；③ 订单管理页支持滑动删除 + 回收站；④ 挂单定义更新为「有毛重但缺皮重」，待开价定义更新为「有毛重+皮重+收款人但缺单价」；⑤ 收款人在所有路径中均为必填项；⑥ 新增回收站页面（buyer-recycle.html），30天保留策略。

1.2 核心业务流程 / 用户旅程地图

  1. 阶段一：开单 — 收购商/游商在收购现场录入商品信息，支持一单一品、拍照留存，底部三段式操作栏（草稿箱/暂存/立即付款）
  2. 阶段二：开单后决策 — 确认开单后选择由谁付款（自己付/选择客户付）
  3. 阶段三：订单管理 — 统一处理中心，管理挂单、待开价、待结算订单，支持滑动删除、回收站、统一详情编辑页
  4. 阶段四：客户功能 — 添加合作收购商为客户，客户收到转来的订单后确认付款或驳回
  5. 阶段五：团队管理 — 收购商默认团队、超管/负责人/开单员三种角色；农户端当前阶段无团队概念；多团队用户可在收购商「我的」页轻量切换当前团队上下文

2. 用户故事详述 (User Stories)

共 13 个用户故事，分布在 5 个阶段中。

阶段一：开单
US-01: 开单页升级 — 一单一品 + 收款人必填 + 三段式底部操作栏
作为 收购商/游商（年龄偏大的操作员）
我希望 开单页只做一件事——录入商品信息，付款决策放在开单后
以便于 降低认知负担，快速完成开单

业务规则与逻辑

  1. 前置条件：用户已登录收购商端，进入开单页面。
  2. 操作流程 (Happy Path)：
    1. 用户选择一级品类（粮食/水果/蔬菜/其他）→ 二级品类。
    2. 点击"添加商品"，录入商品信息：商品名称（必填）、毛重（必填）、皮重（选填）、杂重/扣减（选填）、单价（选填）、备注（选填）。系统自动计算净重和小计。
    3. 一单一品：添加一个商品后，"添加商品"按钮禁用。商品信息平铺展示。
    4. 收款人必填：用户必须选择收款农户（扫码/从合作农户列表选择），该规则适用于立即付款、暂存、欠账等所有路径。
    5. 底部操作栏：左侧「草稿箱」图标入口 + 中间「暂存」按钮 + 右侧「立即付款」按钮。
    6. 点击「立即付款」→ 校验通过 → 生成订单 → 跳转开单完成页（US-05）。
  3. 「暂存」的智能判定规则：
    • 挂单 有毛重但无皮重。保存到订单管理"挂单" Tab。
    • 待开价 有毛重+皮重+收款人，但无单价。保存到"待开价" Tab。
    • 欠账 信息全部完整，但当前不想付款。保存到"待结算" Tab。
    • 系统根据填写情况自动决定暂存后的订单状态，不再通过组合弹层手动选状态。
  4. 「立即付款」校验规则：必须选择品类 ✓ / 必须有一个商品且毛重、皮重已填 ✓ / 必须填写单价 ✓ / 必须选择收款人 ✓
  5. 草稿箱规则：左侧草稿箱图标用于查看和恢复未完成草稿，不承担状态选择逻辑。
  6. 异常处理：未选品类→提示"请先选择开单品类"；未添加商品→提示"请先添加商品"；未填皮重→提示"请先填写皮重"；未填单价→提示"请填写单价"；未选收款人→提示"请选择收款农户"；未填毛重点击暂存→提示"请至少填写毛重"。

场景1: 信息完整立即付款

GIVEN 我已选择品类、添加商品（含毛重、皮重和单价）、选择收款人

WHEN 我点击「立即付款」

THEN 系统校验通过，跳转到开单完成页。

场景2: 有毛重无皮重时暂存为挂单

GIVEN 我已添加商品、填写毛重并选择收款人，但未填皮重

WHEN 我点击「暂存」

THEN 订单保存为 draft 状态，提示"已挂单"，订单出现在订单管理"挂单" Tab。

场景3: 有皮重无单价时暂存为待开价

GIVEN 我已填写毛重、皮重和收款人，但未填单价

WHEN 我点击「暂存」

THEN 订单保存为 pending_price 状态，提示"已保存为待开价"。

US-02: 拍照留存 — 磅单和货物照片作为交易凭证
作为 收购商/游商
我希望 开单时拍照留存磅单、货物照片
以便于 作为交易凭证，减少后续纠纷

业务规则与逻辑

  1. 拍照入口位于商品明细下方、收款对象上方。
  2. 点击"+"按钮，可选择拍照或从相册选择。
  3. 每单最多 9 张照片，以缩略图网格展示。支持预览大图和删除。
  4. 照片不分类、不加水印。照片随订单保存（含所有路径）。
  5. 异常处理：超过 9 张时"+"按钮禁用，提示"最多 9 张照片"；拍照失败时提示"照片获取失败，请重试"。

验收标准

场景1: 成功拍照

GIVEN 我在开单页已添加商品

WHEN 我点击拍照入口并拍摄一张照片

THEN 照片以缩略图展示在拍照区域，计数更新为"(1/9)"。

场景2: 照片随订单保存

GIVEN 我已拍摄 3 张照片并确认开单

WHEN 我在订单详情页查看该订单

THEN 3 张照片在详情页可查看。

场景3: 达到上限

GIVEN 我已拍摄 9 张照片

WHEN 我查看拍照区域

THEN "+"按钮禁用，显示"(9/9)"。

US-03: 暂存功能升级 — 挂单/待开价/欠账三种自动判定路径
作为 收购商/游商
我希望 根据当前填写状态自动进入不同的暂存状态
以便于 后续在订单管理页集中处理

业务规则与逻辑

  • 挂单 draft：已填写毛重和收款人，但没有皮重。说明过磅信息未完整，需后续补皮重。→ 订单管理"挂单" Tab。
  • 待开价 pending_price：已填写毛重 + 皮重 + 收款人，但没有单价。说明重量已确定，价格待补。→ "待开价" Tab。
  • 欠账 unpaid：品类、商品、毛重、皮重、收款人、单价等信息全部完整，但当前不想付款。→ "待结算" Tab。

暂存后：提示"已保存"+ 对应状态名称，开单页表单清空，可继续开下一单。已拍照片随订单一起保存。

约束补充：收款人为所有路径必填项，因此不存在“未选择收款人但能暂存”的合法场景。

验收标准

场景1: 挂单

GIVEN 我已填写毛重并选择收款人，但未填皮重

WHEN 我点击「暂存」

THEN 订单保存为 draft 状态，提示"已挂单"，开单页清空。

场景2: 待开价

GIVEN 我已填写毛重、皮重和收款人，但未填单价

WHEN 我点击「暂存」

THEN 订单保存为 pending_price 状态，提示"已保存为待开价"。

场景3: 欠账

GIVEN 我已填写全部信息（含皮重、单价和收款人）

WHEN 我点击「暂存」

THEN 订单保存为 unpaid 状态，提示"已记欠账"。

阶段二：开单后决策
US-05: 付款人选择页 — 选择由谁付款 + 选择附言
作为 收购商/游商
我希望 确认开单后在一个简洁的页面选择付款人和付款附言
以便于 快速决定付款路径，附言自动传递给银行通道

业务规则与逻辑

  1. 用户在开单页点击「确认开单」且校验通过后，系统生成订单，跳转到付款人选择页。
  2. 页面核心信息为金额（最大视觉重心），下方一行轻摘要（品类 > 商品 · 重量 · 收款人）和订单号。
  3. 附言选择：页面中间有附言选择卡片，默认预选用户设置的默认附言模板（参数已自动填充）。点击可弹出附言选择器底部弹窗，展示所有附言模板及实时预览效果。附言跟着订单走，保存在 order.remark 字段，传递给收银台/银行通道。
  4. 两个操作选项：
    • 我自己付款：保存附言到订单 → 跳转收银台，订单状态变为 paying。
    • 选择客户代付：保存附言到订单 → 弹出客户选择器 → 确认后订单状态变为 pending_client，订单同步到客户端。
  5. 异常处理：用户无客户时→提示"请先添加客户"，提供跳转入口；用户返回上一页→订单不生成，回到开单页（数据保留）。

验收标准

场景1: 选择附言后自己付款

US-09: 团队切换 — 多团队用户切换当前团队上下文
作为 同时属于多个团队的用户
我希望 在「我的」页面切换团队
以便于 在自己的团队和别人的团队间灵活切换

业务规则与逻辑

  1. 入口：收购商「我的」页个人信息下方的轻量当前团队区域，展示当前团队名称 + 角色标签。
  2. 切换展示：仅多团队用户展示切换图标；单团队用户只静态展示团队名称和角色。
  3. 切换交互：点击当前团队区域 → 底部弹出团队列表 → 选择目标团队。
  4. 切换后效果：订单、客户、农户、统计数据全部切换为目标团队的上下文。
  5. 默认团队：首次登录进入自己的团队；切回买方身份时优先恢复上次选择团队，没有历史记录则进入默认团队。
  6. 开单员限制：可见当前团队和角色；不可见团队管理入口，不能自己付款。
  7. 身份切换：买方/卖方切换独立放在顶部右上角，仅双身份用户展示；团队切换不改变买方/卖方身份。

验收标准

场景1: 切换团队

GIVEN 我（小王）属于"小王收购站"（超管）和"张老板收购站"（开单员）

WHEN 我点击轻量当前团队区域，并切换到"张老板收购站"

THEN 界面数据全部切换为张老板团队的上下文，我的角色显示为"开单员"。

US-10: 开单员开单 — 订单自动归属团队
作为 团队开单员
我希望 开单流程和普通收购商一样，但订单归属团队
以便于 替团队超管（货主）收粮，超管可以统一管理和付款

业务规则与逻辑

  1. 开单页：与普通收购商完全一样，无任何区别。
  2. 开单完成页：只显示「选择客户付款」选项（开单员暂不能自己付款）。
  3. 订单归属：开单员开的单自动归属当前团队，超管可在"待结算" Tab 中看到。
  4. 组合按钮：开单员同样可以使用挂单/待开价/欠账功能，订单归属团队。
  5. 超管视角：超管在订单管理中可以看到所有团队成员开的单，通过筛选区分开单人。
  6. 我的页展示：开单员可见当前团队和角色，不展示团队管理入口；农户端不展示团队信息。

验收标准

场景1: 开单员开单

GIVEN 我（小王）在"张老板收购站"团队中，角色为开单员

WHEN 我完成开单 → 在开单完成页选择客户付款 → 选择张老板

THEN 订单状态为待客户付款，张老板在待付款 Tab 中看到该订单（标注"来自：小王"）。

场景2: 开单员欠账

GIVEN 我（小王）在"张老板收购站"团队中

WHEN 我开单后选择「欠账」

THEN 订单进入张老板团队的待结算 Tab，张老板可以看到并处理。
