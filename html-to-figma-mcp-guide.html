为什么这样做

最现实的目标
不追求“HTML 一键变成完美设计系统”，而是把已经完成的页面先快速送进 Figma， 形成一份可以继续调布局、抽组件、补状态、做交付的可编辑稿。
如果你的项目经常是“原型先做 HTML，再回到设计稿统一视觉”，那这套流程最值得沉淀成团队 SOP。
适合
静态原型、GitHub Pages、开发前端页面
不适合
指望直接得到 100% 干净组件库
最佳定位
代码稿 → Figma 中间稿 → UI 精修 → 交付开发

本次实操记录

本次实际执行流程
1. 确认本地服务可访问
使用 `http://127.0.0.1:8877/*.html` 作为抓取来源，先确认目标页面都能返回正确标题。
2. 读取页面语义并准备命名
通过页面标题提炼语义名称，而不是直接使用文件名，避免 Figma 内部可读性差。
3. 新建 Figma 文件与 page
创建新文件后，把默认 page 重命名为 `00 Raw Capture`，作为所有原始抓取结果的唯一入口。
4. 只抓 `.mobile-container`
H5 页本身固定为 `375px`，抓取时指定 `.mobile-container`，避免把桌面浏览器空白区域一起带进 Figma。
5. 逐页串行回收 capture
并发首次尝试容易造成前一个页面还没提交就被后一个页面切走，因此本次最终采用串行提交，更稳定。
6. 清理、重命名、排版
删除误抓的大宽度 frame，仅保留 10 个 `375 x 812` 的有效 frame，并统一改成语义名称。

    标准流程

    流程主线
    准备可抓取页面
    固定 URL、字体、图片、mock 数据和 viewport，确保页面能稳定访问。
    启动本地或线上访问地址
    支持本地服务、GitHub Pages 或已开发的线上预览环境。
    用 MCP 抓取到 Figma
    按“业务流”而不是整站一次性抓，优先抓 8~10 个母版页；H5 页面优先指定 `.mobile-container`。
    UI 在 Figma 中做精修
    优化排版、抽组件、补状态，不需要从头重画整页。
    回流开发
    正式稿进入交付页，后续再结合 MCP / Code Connect 回写代码。
    页面输入
    本地 HTML / GitHub Pages / 线上预览
    推荐每个页面都有独立 URL，避免依赖复杂点击链路。
    抓取粒度
    优先业务流，其次状态页
    登录 首页 市场 发布 收银台 我的 空态 错误态
    建议 viewport
    H5：375 / 393 · PC：1440
    统一 viewport 是为了避免 UI 在 Figma 中拿到不同宽度的混乱稿。H5 若已有手机容器，优先只抓容器本身。

    模板 A · 本次首批 10 页实战 Prompt
    这是本次项目可直接复用的完整批量抓取模板

    适合已存在 Figma 文件时继续向 `00 Raw Capture` 里增量补页
    请继续使用已有的 Figma 文件，把以下页面增量抓取到同一个文件中。 要求： 1. 不要新建文件，继续使用现有文件 2. 所有结果仍然放在 00 Raw Capture 3. H5 页面统一按 375 宽抓取 4. 每个页面一个独立 frame 5. frame 名称使用页面语义名 6. 如果某页首次抓取失败，请自动重试一次，并汇总失败原因 已有 Figma 文件：【替换成文件链接】 待抓取页面： - 【替换成 URL 1】 - 【替换成 URL 2】 - 【替换成 URL 3】
    模板 C · 首次失败页重抓
    适合个别页面长时间 processing 或 unexpected error 时
    请只重抓这一页，不要影响同文件中的其他 frame。 要求： 1. 使用已有 Figma 文件 2. 只重抓这个失败页面 3. 保持放在 00 Raw Capture 4. 如果页面是 H5，请按 375 宽抓取 5. 如果页面有 `.mobile-container`，优先只抓这个容器 6. 成功后请删除误抓或重复的旧 frame，并保留语义名称 失败页面： - 【替换成失败页 URL】 已有 Figma 文件： - 【替换成文件链接】
    模板 D · 补状态页 / 弹窗态
    空态、错误态、校验态、弹窗态建议分批独立补
    请继续使用已有 Figma 文件，为以下状态页补抓独立 frame。 要求： 1. 继续放在 00 Raw Capture 2. 每个状态单独一个 frame 3. 命名采用 “页面语义 + 状态” 方式 4. H5 页面统一抓 375 宽 建议状态： - 空态 - 表单校验错误态 - 成功态 - 弹窗打开态 - 加载态
    模板 E · 指定选择器抓 H5 容器
    适合页面外部有桌面留白或预览壳时
    抓取时请优先只抓 H5 主容器，不要把浏览器桌面背景或额外预览外壳一起抓进 Figma。 要求： 1. 如果页面内存在 `.mobile-container`，只抓 `.mobile-container` 2. 最终 frame 保持 375 宽 3. 如有误抓的大宽度 frame，请自动删除并保留正确版本
    模板 F · 抓 PC 页面
    后续管理后台、看板、PC 收银台可直接用
    请把以下 PC 页面抓到同一个 Figma 文件中，生成可编辑 frame。 要求： 1. PC 页面统一按 1440 宽抓取 2. 每个页面一个独立 frame 3. 放到 00 Raw Capture 4. 名称使用业务语义，不要只用文件名 5. 抓取完成后返回成功列表和失败原因 页面： - 【替换成 PC URL 1】 - 【替换成 PC URL 2】

    后续页面 SOP

    后续每一批页面建议这样抓
    先按业务流分批
    不要一次抓全站。每批保持 4 到 8 页，按一个闭环业务流组织，后续更容易清理和验收。
    仍然先放 00 Raw Capture
    原始抓取页统一留在 `00 Raw Capture`，不要直接在这里做正式视觉稿，后续再拆去 `01 Candidate Components` 和 `02 Polished Screens`。
    命名必须带语义与状态
    例如：`收购商订单列表页`、`收购商订单详情页`、`收购商支付成功态`、`农户收款记录空态`。

    某一页异常时，不要整批推翻；直接使用“失败页重抓模板”，并在成功后清理重复 frame。
    每批都要回报三件事
    新/已有 Figma 文件链接、实际成功页面列表、失败页面及原因。这三项是后续团队协作最低交付标准。

    Figma 文件结构

    00
    Raw Capture
    MCP 抓回来的原始页面，不直接在这里做正式交付。建议保持“一个页面一个 frame + 语义名称”。
    01
    Candidate Components
    从页面里拆出的卡片、按钮、表单、标签、弹窗等候选组件。
    02
    Polished Screens
    UI 精修后的正式页面，适合评审、对齐和后续交付。
    03
    Handoff
    开发交付页、状态说明、关键备注和补充交互说明。

    Codex 接入方式

    三步接入
    1. 添加 Figma MCP
    codex mcp add figma --url https://mcp.figma.com/mcp
    2. 浏览器授权登录
    codex mcp login figma
    3. 查看当前状态
    codex mcp get figma codex mcp list
    判断标准
    已接入未登录
    `Status: enabled` + `Auth: Not logged in`
    已完成可用
    `Status: enabled` + `Auth: OAuth`
    如果你已经把 MCP 装进 Codex，但当前这个会话里还没直接出现新的 figma 工具句柄，通常重新开一个 Codex 会话就能加载到最新配置。

    质量检查

    页面宽度是否正确，移动端和 PC 是否混宽。
    文本是否仍为可编辑文本，是否被错误转成图层。
    图片、图标、按钮、卡片是否还能被进一步拆组件。
    空态、错误态、弹窗态是否已经单独成 frame；失败重抓后是否清理了重复 frame。
    10

    已知边界

    它能做的
    • 快速把 live UI 送进 Figma
    • 节省重画成本
    • 让 UI 在此基础上继续精修
    • 适合沉淀组件和正式交付页
    它做不到的
    • 不能保证 100% 自动得到最优组件体系
    • 复杂动画、伪元素和特殊定位可能仍需人工修
    • 需要你提前准备好可访问、可复现的页面状态
    • 后续若追求生产级代码，仍建议补上组件化整理和 Code Connect
